# 特性原子能力提取方法论

## 一、定义

特性原子能力是从SE文档中提取的系统对外提供的**最小、独立、完整、不可再拆分**的业务能力单元，屏蔽技术实现细节，支撑用例设计、规则沉淀、覆盖分析、精准回归。

## 二、核心原则

| 原则 | 说明 |
|------|------|
| 只抽业务，不抽技术 | 屏蔽技术实现细节 |
| 最小不可拆，单一职责 | 一个原子只做一件事 |
| 无技术词、无实现、无架构 | 纯业务语言描述 |
| 可独立测试、可映射IBO | 支持测试用例生成 |
| 全链路可追溯、可治理 | 需求→特性→规则→用例 |

### 2.1 必须抽取的内容

- **各类业务校验**：唯一性、格式、范围、权限、关联依赖、业务规则校验、重复性拦截
- **后台自动业务处理**：自动状态变更、自动补全、自动计算、数据清洗、格式标准化
- **数据治理类动作**：数据关联绑定、关系解绑、历史归档、数据快照、台账维护
- **安全 & 审计 & 管控**：数据脱敏、访问拦截、操作留痕、权限控制、数据隔离
- **跨模块联动**：数据同步、业务消息流转、上下游联动、状态联动更新
- **异常兜底能力**：容错处理、数据修复、规则兜底、冲突消解

### 2.2 绝对不抽的内容

严禁提取技术实现细节，以下词汇及概念直接过滤：

- 接口定义、字段、数据库表、缓存、MQ、微服务、事务、线程、部署、硬件、算法代码、参数入参出参、技术架构
- 禁止词汇：后台、接口、服务、数据库、字段、缓存、SQL、定时任务

## 三、提取字段规格

### 字段定义与提取方式

| 字段 | 类型 | 提取方式 | 说明 |
|-----|------|---------|------|
| atom_id | str | 模型生成 | 格式为 `feat_atom_{1,2,3,...}`，编号从1开始递增 |
| atom_name | str | 模型生成 | 对原子能力的一句话概括，简洁明了 |
| actor | str | 模型提取 | 执行该原子能力的角色（系统、用户、管理员等） |
| subject | str | 模型提取 | 关联主体，该能力作用于哪个业务对象 |
| entity | str | 模型提取 | 主要业务实体名词 |
| entity_attrs | list[str] | 模型提取 | 该实体的关键属性列表 |
| behavior | str | 模型提取 | 业务行为动词，描述做什么（校验、计算、变更等） |
| pre_condition | str | 模型提取 | 执行前必须满足的业务条件 |
| expected_result | str | 模型提取 | 预期的输出或状态变更 |
| ref_l3_rules | list[dict] | 模型提取 | 关联L3规则，参考《原子规则提取方法论》 |

### 字段语义区分

- **actor（执行角色）**：谁执行这个行为？如"系统"、"用户"、"管理员"
- **subject（关联主体）**：这个能力与哪个业务过程相关？如"用户注册"、"订单处理"
- **entity（业务实体）**：主要涉及的实体对象，如"用户"、"订单"、"商品"
- **entity_attrs（实体属性）**：该实体的关键属性，如用户名、手机号、订单号
- **behavior（业务行为）**：具体的业务动作，如"校验唯一性"、"计算积分"、"变更状态"

## 四、提取流程

1. **扫描全文**：通读SE文档，识别所有特性原子能力
2. **逐个提取**：针对每个识别出的原子能力：
   - 生成atom_id（feat_atom_1, feat_atom_2, ...）
   - 总结atom_name
   - 按字段规格提取剩余字段

## 六、质量检查清单

提取完成后，逐项检查每个原子能力：

- [ ] **纯业务语言**：描述的是业务能力，不包含任何技术术语
- [ ] **最小不可拆**：是独立、最小的业务单元，不可再拆分
- [ ] **可独立测试**：有明确的前置条件和预期结果，可单独测试
- [ ] **字段完整**：所有必填字段都有值，ref_l3_rules可为空数组
- [ ] **语义正确**：actor/subject/entity/behavior语义清晰不混淆
- [ ] **编号连续**：atom_id从1开始连续递增
